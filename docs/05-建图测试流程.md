# 建图测试流程

这份文档只回答一个问题: 当你怀疑底盘里程计、下位机航向、EKF 或雷达外参影响建图质量时，现场应该怎么测，才能得到可复现结论。

目标不是“边试边感觉”，而是把每次测试变成可比较的 A/B 记录。

## 1. 测试前提

开始前先满足这几个条件:

- 雷达健康检查通过
- 底盘可以正常受控
- 使用统一入口 `robot.sh`
- 不要一开始就测 `precision`

推荐先从 `quality` 档开始，因为它更适合作为对照基线。

## 2. 本轮测试要回答什么

一次测试最多回答一个主问题。

常见问题有三类:

1. 下位机 `yaw_est` 直接用于建图，是否比桥接积分 `w_est` 更好
2. `--ekf-base` 是正收益还是负收益
3. 当前问题更像底盘航向误差，还是雷达外参误差

如果你一次同时改 `yaw`、`EKF`、SLAM 参数、雷达外参，最后无法判断谁起作用。

## 3. 推荐测试顺序

### 第一轮: 先验证下位机航向

先做这两组:

```bash
./robot.sh mapping lidar --real-base
./robot.sh mapping lidar --real-base --base-use-status-yaw
```

这轮只回答:

- 直接使用下位机 `yaw_est` 是否改善角点和矩形闭环

### 第二轮: 再验证 EKF

只有当第一轮做完后，才建议做 EKF 对照:

```bash
./robot.sh mapping lidar --real-base --ekf-base
./robot.sh mapping lidar --real-base --ekf-base --base-use-status-yaw
```

这轮只回答:

- 当前 EKF 配置是在帮忙，还是在把同源数据又滤坏一遍

### 第三轮: 最后才碰精细档

如果 `quality` 已经稳定，再测:

```bash
./robot.sh mapping lidar precision --real-base --base-use-status-yaw
```

如果 `quality` 还不稳，不要先测 `precision`。

## 4. 每组测试的标准路线

每组都走完全一样的路线，不能这组随便开，那组贴墙走。

推荐路线:

1. 原地静止 10 秒
2. 直线前进 2 米
3. 直线后退 2 米
4. 原地左转 90 度，停 2 秒
5. 再左转 90 度
6. 贴墙走矩形一圈
7. 回到起点附近静止 5 秒

如果场地有限，至少保留:

1. 静止
2. 直线
3. 两次 90 度转向
4. 一圈矩形

## 5. 每组测试怎么录

先开一个终端启动系统，再开另一个终端录包。

### 基线组

终端 1:

```bash
cd /home/robot/ros2_ws/launch_scripts
./robot.sh mapping lidar --real-base
```

终端 2:

```bash
cd /home/robot/ros2_ws/launch_scripts
./record_mapping_test.sh baseline --notes "./robot.sh mapping lidar --real-base"
```

### 下位机航向组

终端 1:

```bash
./robot.sh stop
./robot.sh mapping lidar --real-base --base-use-status-yaw
```

终端 2:

```bash
./record_mapping_test.sh status-yaw --notes "./robot.sh mapping lidar --real-base --base-use-status-yaw"
```

### EKF 组

终端 1:

```bash
./robot.sh stop
./robot.sh mapping lidar --real-base --ekf-base
```

终端 2:

```bash
./record_mapping_test.sh ekf --notes "./robot.sh mapping lidar --real-base --ekf-base"
```

### EKF + 下位机航向组

终端 1:

```bash
./robot.sh stop
./robot.sh mapping lidar --real-base --ekf-base --base-use-status-yaw
```

终端 2:

```bash
./record_mapping_test.sh ekf-status-yaw --notes "./robot.sh mapping lidar --real-base --ekf-base --base-use-status-yaw"
```

## 6. `record_mapping_test.sh` 会做什么

脚本会自动:

- 创建带时间戳的测试目录
- 生成 `TEST_RECORD.md`
- 生成 `metadata.json`
- 录制这些话题:
  - `/scan`
  - `/odom`
  - `/imu/data`
  - `/tf`
  - `/tf_static`
  - `/odometry/filtered`

注意:

- 没开 `--ekf-base` 时，`/odometry/filtered` 没数据是正常的
- 没有启用 IMU 发布时，`/imu/data` 可能为空或没有有效更新

## 7. 实时量化监视

如果你不想只看 RViz，可以再开第三个终端跑实时指标监视:

```bash
cd /home/robot/ros2_ws/launch_scripts
./monitor_mapping_metrics.py --csv-out ~/ros2_test_records/live_metrics.csv
```

它会持续输出这些量:

- `/odom` 的 `x y yaw vx wz`
- `/imu/data` 的 `yaw gz ax ay az`
- `/odometry/filtered` 的 `x y yaw vx wz`
- `odom - imu` 的航向差
- `filtered - imu` 的航向差
- 当前静止段的位移漂移和航向漂移

如果你在“静止 10 秒”这一步看到:

- `static.active=true`
- `static.odom_drift` 持续变大
- `static.imu_yaw_drift` 持续变大

那就说明问题不是地图渲染，而是底盘数据流自己在漂。

## 8. 每组重点看什么

不要同时看太多指标。先看这五项:

- 静止时地图或 `/odom` 会不会自己漂
- 直线时车是否向一侧偏
- 两次 90 度转向后，角点是否接近直角
- 矩形四条边是否平直
- 回到起点后，地图是否能基本对齐

如果现场还有精力，再补充:

- `/odom` 和实际车体方向是否一致
- 转弯时地图是否出现“边走边扭”
- 停车后地图是否继续缓慢转动

## 9. 测完后怎么写记录

每组测试结束后，打开脚本生成的 `TEST_RECORD.md`，至少补这些内容:

- 本次实际启动命令
- 静止漂移是否存在
- 直线横漂是否明显
- 90 度角是否正常
- 矩形是否闭合
- 与上一组相比，是更好、持平还是更差

不要只写“感觉还行”。至少写成这种粒度:

- 静止 10 秒，地图基本不转
- 直线 2 米，右偏约 8 cm
- 两次 90 度后，墙角有轻微钝化
- 回到起点后，闭环误差约 15 cm

## 10. 怎么根据结果判断问题归属

### 情况 A

- 基线差
- `--base-use-status-yaw` 明显变好

结论:

- 问题主要在桥接自己积分 `w_est`
- 后续应优先使用下位机 `yaw_est`

### 情况 B

- 基线好
- `--ekf-base` 变差

结论:

- 当前 EKF 是负收益
- 先不要在建图链路里启用 EKF

### 情况 C

- 四组都差不多歪

结论:

- 更像雷达外参或底盘几何标定问题
- 先查 `base_link -> laser_frame`、轮距、速度缩放

### 情况 D

- 静止时都自己漂

结论:

- 优先查 `yaw_est` 稳定性、`w_est` 静止偏置、IMU 有效性

## 11. 推荐的最小测试集

如果你今天只做最小验证，只做两组:

```bash
./robot.sh mapping lidar --real-base
./robot.sh mapping lidar --real-base --base-use-status-yaw
```

这是当前最有价值的一轮，因为它能最快回答:

- 上位机到底该不该直接信下位机当前融合航向

## 12. 什么时候再继续调参数

只有在 A/B 结果清楚以后，才进入下一步:

1. 先定航向来源
2. 再定是否保留 EKF
3. 再校雷达外参
4. 最后才调 `slam_toolbox` 门限

顺序反了，测试结果通常会失真。

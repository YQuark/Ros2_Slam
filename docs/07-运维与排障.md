# 运维与排障

这份文档把日常检查和排障动作收口到一起，避免每次都去翻脚本。

## 1. 常用命令

系统诊断:

```bash
cd /home/robot/ros2_ws/launch_scripts
./robot.sh doctor
```

停止所有 ROS2 会话:

```bash
./robot.sh stop
```

雷达健康检查:

```bash
./robot.sh check lidar
```

## 2. 优先检查顺序

1. `doctor`
2. 设备是否接线正确
3. 串口是否被占用
4. 雷达是否有 `/scan`
5. 底盘是否有 `/odom`
6. RViz 是否异常

## 3. 雷达常见问题

- `/scan` 数量为 0
- `Check Sum` 或 `Failed to get scan`
- 雷达健康检查不通过

优先动作:

- 换 USB 线
- 检查供电
- 检查是否有重复启动的雷达进程
- 再跑 `./robot.sh check lidar`

如果 RViz 里 `Fixed Frame=base_link`，只看 `LaserScan` 时出现“真实左侧物体显示在右侧”，优先按下面顺序排查：

1. 看 `src/robot_bringup/config/ydlidar_X2_mapping.yaml` 是否为 `inverted: true`
2. 用 `./robot.sh sensor lidar` 只验证雷达，不要混入建图或导航
3. 需要临时 A/B 时，用 `./robot.sh mapping lidar --real-base --lidar-inverted` 或 `--no-lidar-inverted`
4. 仍异常时再测试 `--lidar-reversion` 和 `--lidar-yaw-*`

这类左右镜像问题优先属于 `/scan` 相对 `base_link` 的手性错误，不要先改 `BASE_USE_STATUS_YAW`、`BASE_ODOM_FEEDBACK_SOURCE` 或 `BASE_ODOM_ANGULAR_SIGN`。

## 4. 底盘常见问题

- 串口识别成错误设备
- 底盘不响应 `/cmd_vel`
- `/odom` 不更新

优先动作:

- 看 `base_mode`
- 看 `base_port`
- 看底盘是否上电
- 看串口是否被占用

## 5. RViz 常见问题

当前工程在 RViz 上使用了软件渲染策略来规避部分 OpenGL 兼容问题。
如果地图层显示异常，优先判断是渲染问题还是 SLAM 本身问题。

判断方式:

- `/map` 是否仍在发布
- `/scan` 是否仍在更新
- `slam_toolbox` 是否仍在运行

## 6. 话题级检查

```bash
ros2 topic list | grep -E '^/scan$|^/camera/scan$|^/map$|^/odom$|^/odometry/filtered$|^/imu/data$|^/cmd_vel$'
```

## 7. 节点级检查

```bash
ros2 node list
```

如果节点存在但数据没动，问题通常在串口、传感器或参数文件。

## 8. 当前默认设备约定

- 底盘默认串口: `/dev/ttyUSB1`
- 雷达默认串口: `/dev/ttyUSB0`
- 默认雷达建图参数: `src/robot_bringup/config/ydlidar_X2_mapping.yaml`

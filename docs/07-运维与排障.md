# 运维与排障

本文档汇总日常检查命令、常见故障排查流程和系统监控方法。

## 1. 入口约定

- 用户和运维入口：`launch_scripts/robot.sh`
- 工程编排入口：`src/robot_bringup/launch/system.launch.py`
- 默认导航流程：`./robot.sh navigation ...`
- 两阶段导航：仅作为回退和诊断模式

## 2. 常用命令

```bash
cd /home/robot/ros2_ws/launch_scripts

# 系统诊断
./robot.sh doctor

# 停止当前会话
./robot.sh stop

# 健康检查
./robot.sh check lidar         # 雷达健康检查
./robot.sh check chassis       # 底盘健康检查
./robot.sh check tf            # TF 树检查
./robot.sh check mapping       # 建图链路检查
./robot.sh check navigation    # 导航链路检查

# 单独启动
./robot.sh sensor lidar        # 仅雷达
./robot.sh base                # 仅底盘桥接
./robot.sh teleop              # 键盘遥控
```

## 3. check 子命令

| 目标 | 命令 | 检查内容 |
|------|------|----------|
| `lidar` | `./robot.sh check lidar` | 雷达串口、权限、驱动、`/scan` 频率 |
| `chassis` | `./robot.sh check chassis` | 底盘串口、权限、桥接连通性 |
| `tf` | `./robot.sh check tf` | TF 树完整性、关键 TF 是否存在 |
| `mapping` | `./robot.sh check mapping` | `/scan`、`/map`、`slam_toolbox` 状态 |
| `navigation` | `./robot.sh check navigation` | AMCL、Nav2、`/plan`、`/cmd_vel` 状态 |

## 4. 故障排查优先级

1. `./robot.sh doctor` — 系统级诊断
2. 设备接线和供电
3. 串口是否冲突或被旧进程占用
4. 雷达 `/scan` 是否正常
5. 底盘 `/odom` 是否正常
6. RViz 是否只是显示异常
7. 导航前是否已完成 `2D Pose Estimate`

## 5. 导航异常分流

| 现象 | 排查方向 |
|------|----------|
| 定位没收敛 | 先执行 `2D Pose Estimate` |
| 激光与地图不重合 | 检查激光手性和外参 |
| `2D Goal Pose` 后无反应 | `./robot.sh check navigation` |
| 有 `/plan` 但车不走 | 检查 `/cmd_vel`、`/local_plan` 和底盘 |
| 路径太保守或拐弯生硬 | 确认 footprint 与实物一致，检查 `/local_plan` |
| Nav2 起不来 | 回退到 `--localization-only` / `--nav2-only` |

两阶段回退：

```bash
./robot.sh navigation --real-base --ekf-base --localization-only
# RViz 完成 2D Pose Estimate 后
./robot.sh navigation --nav2-only
```

## 6. 雷达常见问题

| 问题 | 排查步骤 |
|------|----------|
| `/scan` 数量为 0 | 检查 USB 线和供电；`./robot.sh check lidar` |
| `Check Sum` 或 `Failed to get scan` | 换 USB 线；检查串口权限 |
| 左右镜像 | 测试 `--lidar-inverted` / `--no-lidar-inverted`；检查 `ydlidar_x2.yaml` 中 `inverted` |

激光左右镜像不应通过修改 odom 符号链来修复。

## 7. 底盘常见问题

| 问题 | 排查步骤 |
|------|----------|
| 串口识别成错误设备 | 检查 `excluded_ports` 配置；使用 udev 规则区分 |
| 底盘不响应 `/cmd_vel` | 检查 `base_mode`、`base_port`、底盘上电 |
| `/odom` 不更新 | 检查 STATUS 帧是否返回；`./robot.sh check chassis` |
| 运动方向不一致 | 检查 `odom_angular_sign` 参数 |
| 底盘处于非闭环模式 | 日志会每 2 秒告警一次 |

## 8. RViz 常见问题

地图层显示异常时先区分是渲染问题还是数据问题：

- `/map` 是否仍在发布
- `/scan` 是否仍在更新
- `slam_toolbox` 是否仍在运行

## 9. 话题和节点检查

```bash
ros2 topic list | grep -E '^/scan$|^/map$|^/odom$|^/odometry/filtered$|^/imu/data$|^/cmd_vel$'
ros2 node list
```

节点存在但数据不动时，问题通常在串口、传感器、参数或状态机前置条件。

导航不响应目标时：

```bash
./robot.sh check navigation
ros2 topic echo /goal_pose --once
ros2 topic echo /plan --once
ros2 topic echo /local_plan --once
ros2 topic hz /cmd_vel
```

## 10. 温度和降频监控

树莓派 4B 在 80°C 触发 CPU 降频，影响 SLAM 和导航性能。

```bash
# 查看 CPU 温度
vcgencmd measure_temp

# 查看是否已降频
vcgencmd get_throttled

# 持续监控
watch -n 2 vcgencmd measure_temp
```

`get_throttled` 返回值含义：

| 位 | 含义 |
|----|------|
| 0 | 当前已降压 |
| 1 | 当前已降频 |
| 16 | 过去曾降压 |
| 17 | 过去曾降频 |

返回 `0x0` 表示正常。返回 `0x50005` 表示当前正在降频降压。

## 11. systemd 服务管理

如已安装 systemd 服务：

```bash
# 安装服务
sudo /home/robot/ros2_ws/launch_scripts/install_service.sh

# 查看状态
sudo systemctl status robot

# 启动/停止/重启
sudo systemctl start robot
sudo systemctl stop robot
sudo systemctl restart robot

# 查看日志
sudo journalctl -u robot -f
```

## 12. 当前默认设备约定

- 底盘：`port:=auto`（自动探测 CP210x）
- 雷达：`/dev/ydlidar`（udev 规则）或自动探测
- 雷达参数：`src/robot_bringup/config/ydlidar_x2.yaml`
- 导航参数：`src/robot_bringup/config/nav2_params_robot.yaml`
- 默认地图：`~/ros2_maps/latest.yaml`

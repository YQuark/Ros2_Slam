# 底盘与串口桥接

这份文档描述 `stm32_robot_bridge` 在整个工程里的位置。

## 1. 职责

`stm32_robot_bridge` 负责把上位机的运动控制和下位机的反馈串起来。

它关心的不是地图，也不是雷达，而是这条控制闭环:

- 输入: `/cmd_vel`
- 输出: 底盘动作
- 反馈: `/odom`、`odom->base_link`
- 可选反馈: `/imu/data`

## 2. 启动入口

统一入口:

```bash
cd /home/robot/ros2_ws/launch_scripts
./robot.sh base
```

如果需要指定串口:

```bash
./robot.sh base --port /dev/ttyUSB1
```

如果要在建图或导航中启用底盘 EKF:

```bash
./robot.sh mapping lidar --real-base --ekf-base
./robot.sh navigation /home/robot/ros2_maps/my_map.yaml --real-base --ekf-base
```

如果希望 `/odom.pose.yaw` 直接采用下位机 `yaw_est`，当前可用入口有两类:

```bash
./robot.sh base --use-status-yaw
./robot.sh navigation /home/robot/ros2_maps/my_map.yaml --real-base --base-use-status-yaw
```

建图模式目前不提供 `--base-use-status-yaw` CLI，若要做 A/B，可用环境变量：

```bash
BASE_USE_STATUS_YAW=false ./robot.sh mapping lidar --real-base
BASE_USE_STATUS_YAW=true ./robot.sh mapping lidar --real-base
```

## 3. 关键参数

在 `system.launch.py` 中与底盘相关的参数主要有:

- `base_mode`
- `base_port`
- `base_baudrate`
- `base_max_linear`
- `base_max_angular`
- `base_publish_tf`
- `base_fusion_mode`
- `base_imu_topic`
- `base_imu_frame_id`
- `base_use_status_yaw`

## 4. 运行模式

- `base_mode:=real` 使用真实底盘
- `base_mode:=fake` 使用虚拟底盘
- `base_mode:=none` 不启用底盘

## 5. 自动探测

当前工程默认会尝试自动识别底盘口。
如果识别成功，脚本和 launch 会把端口透传给桥接节点。
如果没有识别成功，`auto` 模式会持续重试，而不是直接退出。

补充说明:

- 串口探测不再只按 CP210x 设备名猜测，会主动向候选串口发送 `GET_STATUS` 进行探测
- 串口连通后桥接会先尝试把底盘切到 `MODE_CLOSED_LOOP` 并发送零速同步，再开始正常收发
- 桥接长期状态输入以二进制 `GET_STATUS` 为主，ASCII `TEL` 只用于调试和人工核对字段
- 下位机导出的 `yaw_est / gz / raw_accel` 已经按当前固件安装映射转换到车体系，桥接不要再二次旋转
- `/imu/data` 默认 `frame_id=base_link` 依赖上面这条前提；当前 EKF 只融合 yaw 和 z 轴角速度，不把原始加速度当平移里程计

## 6. 你在现场应该怎么判断桥接是否正常

- `/cmd_vel` 有发布者
- `stm32_robot_bridge` 节点存在
- `/odom` 在更新
- 机器人运动和上位机命令一致
- 启用 `--ekf-base` 时 `/imu/data` 和 `/odometry/filtered` 也应持续更新

## 7. 常见问题

- 串口口位变化
- 底盘未上电
- 串口被旧进程占用
- 指令发出但没有反馈
- `base_mode` 和真实接线不一致
- 底盘处于非闭环模式，导致 `/cmd_vel` 被下位机拒绝

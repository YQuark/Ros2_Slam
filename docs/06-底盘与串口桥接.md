# 底盘与串口桥接

这份文档描述 `stm32_robot_bridge` 在整个工程里的位置。

## 1. 职责

`stm32_robot_bridge` 负责把上位机的运动控制和下位机的反馈串起来。

它关心的不是地图，也不是雷达，而是这条控制闭环:

- 输入: `/cmd_vel`
- 输出: 底盘动作
- 反馈: `/odom`、`odom->base_link`

## 2. 启动入口

统一入口:

```bash
cd /home/robot/ros2_ws/launch_scripts
./robot.sh base
```

如果需要指定串口:

```bash
./robot.sh base --port /dev/ttyUSB0
```

## 3. 关键参数

在 `system.launch.py` 中与底盘相关的参数主要有:

- `base_mode`
- `base_port`
- `base_baudrate`
- `base_max_linear`
- `base_max_angular`
- `base_publish_tf`

## 4. 运行模式

- `base_mode:=real` 使用真实底盘
- `base_mode:=fake` 使用虚拟底盘
- `base_mode:=none` 不启用底盘

## 5. 自动探测

当前工程默认会尝试自动识别底盘口。
如果识别成功，脚本和 launch 会把端口透传给桥接节点。
如果没有识别成功，`auto` 模式会持续重试，而不是直接退出。

## 6. 你在现场应该怎么判断桥接是否正常

- `/cmd_vel` 有发布者
- `stm32_robot_bridge` 节点存在
- `/odom` 在更新
- 机器人运动和上位机命令一致

## 7. 常见问题

- 串口口位变化
- 底盘未上电
- 串口被旧进程占用
- 指令发出但没有反馈
- `base_mode` 和真实接线不一致

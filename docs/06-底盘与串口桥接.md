# 底盘与串口桥接

这份文档描述 `stm32_robot_bridge` 在整个工程中的位置，以及它和建图、导航主流程的关系。

## 1. 职责

`stm32_robot_bridge` 只关心底盘控制闭环，不关心地图和雷达。

- 输入：`/cmd_vel`
- 输出：底盘动作
- 反馈：`/odom`、`odom -> base_link`
- 可选反馈：`/imu/data`

## 2. 推荐入口

统一入口：

```bash
cd /home/robot/ros2_ws/launch_scripts
./robot.sh base
```

指定串口：

```bash
./robot.sh base --port /dev/ttyUSB1
```

在主业务里，底盘通常不单独启动，而是被下面这些正式流程带起：

```bash
./robot.sh mapping lidar --real-base --ekf-base
./robot.sh navigation --real-base --ekf-base
```

## 3. 常用控制相关参数

- `--port` / `BASE_PORT`
- `--use-status-yaw` / `BASE_USE_STATUS_YAW`
- `BASE_STATUS_YAW_MODE`
- `BASE_STATUS_YAW_JUMP_REJECT_DEG`
- `BASE_ODOM_FEEDBACK_SOURCE`
- `BASE_WHEEL_TRACK_WIDTH`
- `BASE_ODOM_ANGULAR_SCALE`
- `BASE_ODOM_ANGULAR_SIGN`

如果只是正常运行，优先保持默认值。需要 A/B 时，再用环境变量或 CLI 覆写。

## 4. 当前协议和运行假设

- 串口默认参数：`115200 8N1`
- 上位机要求下位机进入 `MODE_CLOSED_LOOP`
- 状态包优先使用二进制 `GET_STATUS`
- `/odom.twist` 默认使用状态包里的 `v_est / w_est`
- `use_status_yaw:=true` 时，只在下位机 `imu_valid=1` 的状态帧中使用 `yaw_est`
- EKF 只把 `/imu/data` 当作姿态与角速度输入，不把原始加速度当平移里程计
- 当前统一默认时序：`cmd_timeout=0.25s`、`drive_keepalive_sec=0.10s`
- 时序对齐原则：
  - 上位机 `control_hz=20Hz` 对齐下位机 `CTRL_PERIOD_MS=20ms`
  - 上位机 `status_hz=20Hz` 对齐下位机主循环的 `GET_STATUS` 处理窗口
  - 上位机 `drive_keepalive_sec=0.10s` 对齐 ESP/PC 侧 `100ms` 控制刷新节奏
  - 上位机 `cmd_timeout=0.25s` 配合 `20Hz` 控制循环，保证失去 `/cmd_vel` 后最晚在约 `0.30s` 内发出显式零速，不反向覆盖下位机 `CMD_TIMEOUT_MS=300ms`

## 5. 自动探测和保护行为

- `port:=auto` 时会优先扫描 CP210x 候选串口
- 底盘探测会排除已知雷达串口
- 串口连通后桥接会先发闭环模式和零速同步
- 状态过期时 `/odom` 不再回退到 `/cmd_vel` 命令速度，而是保持位姿、速度置零并提高协方差

## 6. 现场应该怎么看桥接是否正常

- `/cmd_vel` 有发布者
- `stm32_robot_bridge` 节点存在
- `/odom` 在更新
- 机器人运动方向和上位机命令一致
- 启用 `--ekf-base` 时 `/imu/data` 和 `/odometry/filtered` 也持续更新

## 7. 常见问题

- 串口口位变化
- 底盘未上电
- 串口被旧进程占用
- 底盘处于非闭环模式，导致 `/cmd_vel` 被下位机拒绝
- `/odom` 有话题但反馈不随真实运动变化

如果是主业务异常，先回到 [07-运维与排障](./07-运维与排障.md)；如果是链路深挖，再看 [05-导航拆解调试](./05-导航拆解调试.md)。

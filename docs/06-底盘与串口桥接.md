# 底盘与串口桥接

本文档描述 `stm32_robot_bridge` 包的内部实现、v1 协议细节和运行时行为。协议帧格式和命令定义的权威来源见 [04-串口桥接协议](./04-串口桥接协议.md)。

## 1. 职责

`stm32_robot_bridge` 仅负责底盘控制闭环：

- 输入：订阅 `/cmd_vel`（`geometry_msgs/Twist`）
- 输出：通过串口发送 SET_VELOCITY 控制帧
- 反馈：从 STATUS 帧解析轮速、编码器、电池、电流、IMU，发布 `/odom`、`/tf` 和辅助话题

不参与地图构建、雷达数据处理或路径规划。

## 2. 启动入口

```bash
# 单独启动
./robot.sh base

# 指定串口（默认 auto 自动探测）
./robot.sh base --port /dev/stm32_chassis

# 由建图或导航流程自动带起
./robot.sh mapping lidar --real-base --ekf-base
./robot.sh navigation --real-base --ekf-base
```

## 3. 代码结构

### Protocol 类（静态工具类）

文件：`src/stm32_robot_bridge/stm32_robot_bridge/bridge_node.py`

```python
class Protocol:
    @staticmethod
    def build_frame(cmd: int, payload: bytes) -> bytes
```

构造完整 v1 帧：`0xA5 0x5A + length(1B) + cmd(1B) + payload + checksum8(1B)`。校验算法：`(length + cmd + sum(payload)) & 0xFF`。

```python
    @staticmethod
    def find_frame(buf: bytearray) -> Tuple[Optional[Tuple[int, bytes]], int]
```

在缓冲区中搜索有效帧。返回 `((cmd, payload), bytes_consumed)` 或 `(None, search_position)`。

搜索逻辑：

1. 逐字节扫描 `0xA5 0x5A` 帧头
2. 读取 length 和 cmd
3. 计算帧结束位置：`header(4) + payload(length) + checksum(1)`
4. 缓冲区数据不足时返回 `(None, header_position)`，等待更多数据
5. 校验 checksum，匹配则返回帧内容
6. 校验失败则跳过当前帧头继续搜索

### STM32Bridge 类（ROS2 节点）

节点名：`stm32_bridge`

#### 初始化流程

1. 声明参数（22 个，详见 [04-串口桥接协议](./04-串口桥接协议.md) 第 8 节）
2. 创建发布者：`/odom`、`/tf`、`/imu/data`（可选）、`/chassis/status`、`/battery_state`、`/motor/left_current`、`/motor/right_current`
3. 创建订阅者：`/cmd_vel`
4. 注册定时器：`control_loop`（20Hz）、`status_poll`（20Hz）
5. 调用 `ensure_serial()` 尝试打开串口

#### 串口自动探测

`resolve_port()` 方法：

1. 若 `port` 参数非 `auto`，直接使用指定路径
2. 调用 `detect_cp210x_ports(excluded_ports)` 扫描 CP210x 设备
3. 对每个候选端口调用 `probe_port_for_status()` 验证连通性

`probe_port_for_status(port)` 方法：

1. 打开串口，设置 DTR/RTS 为低
2. 等待 `startup_settle_sec`（0.20s）
3. 发送 CMD_STATUS 请求（最多 3 次）
4. 每次等待 0.35s，使用 `Protocol.find_frame()` 搜索响应
5. 收到 47 字节 STATUS 响应则返回 `True`

`detect_cp210x_ports()` 函数：

- 扫描 `/dev/serial/by-id/*CP210*`、`*cp210*`、`*Silicon_Labs*`、`*USB*UART*`
- 扫描 `/dev/ttyUSB*`，通过 `/sys/class/tty/*/driver` 和 USB vendor/product ID 识别 CP210x
- 排除 `excluded_ports` 中的设备
- 仅有一个 `/dev/ttyUSB*` 时直接使用（回退策略）

#### 串口管理

`ensure_serial()` 方法：

- 串口已打开时直接返回 `True`
- 未打开时以 `serial_open_retry_sec`（2.0s）间隔重试
- 打开成功后：清空缓冲区、发送零速同步（3 次）、发送 STATUS 请求

`close_serial(reason)` 方法：

- 关闭串口、清空缓冲区、重置驱动状态
- 记录告警日志

#### 控制循环（20Hz）

`control_loop()` 方法执行以下步骤：

1. `ensure_serial()` — 确保串口已打开
2. `read_serial_frames()` — 读取并处理串口数据
3. 检查 `/cmd_vel` 超时：
   - 超时（>0.25s）：发送零速
   - 未超时：使用目标速度
4. 编码驱动帧：`struct.pack('<ffBB', vx, wz, 1, MODE_CLOSED_LOOP)`
5. 保活逻辑：速度变化或距上次发送超过 `drive_keepalive_sec`（0.10s）时重发
6. `publish_odom(now)` — 发布里程计

#### 状态轮询（20Hz）

`status_poll()` 方法：

1. `ensure_serial()`
2. `read_serial_frames()`
3. 发送 CMD_STATUS 请求

#### 帧处理

`read_serial_frames()` 方法：

1. 从串口读取可用数据到 `rx_buf`
2. 循环调用 `Protocol.find_frame()` 搜索有效帧
3. 找到帧则调用 `handle_frame(cmd, payload)`
4. 校验失败的字节被丢弃，防止缓冲区无限增长

`handle_frame(cmd, payload)` 方法（仅处理 CMD_STATUS=0x81）：

1. 验证 payload 长度 ≥ 47 字节
2. 解析字段：

```python
struct.unpack_from('<ff i i f f f h h h h h h I B', payload, 0)
# left_speed(f32) right_speed(f32)
# left_enc(i32) right_enc(i32)
# battery_voltage(f32) left_current(f32) right_current(f32)
# accel_x/y/z(i16) gyro_x/y/z(i16)
# error_flags(u32) control_mode(u8)
```

3. IMU 换算：
   - 加速度：`real = raw * 16.0 * 9.80665 / 32768`（m/s²）
   - 角速度：`real = raw * 2000.0 * π / (180 * 32768)`（rad/s）
4. 计算轮速里程计（见第 4 节）
5. 发布辅助话题

#### 里程计计算

`publish_odom(now)` 方法：

1. 检查 STATUS 帧新鲜度：`age ≤ status_timeout`（0.75s）
2. 新鲜时使用反馈速度，过期时速度置零
3. 应用死区：线速度 <0.01 m/s、角速度 <0.03 rad/s 时置零
4. 积分位姿：

```text
vx = (left_speed + right_speed) / 2 * odom_linear_scale
wz = (right_speed - left_speed) / wheel_track_width * odom_angular_scale * odom_angular_sign

yaw_mid = yaw + wz * dt / 2
x += vx * cos(yaw_mid) * dt
y += vx * sin(yaw_mid) * dt
yaw += wz * dt
```

5. 发布 `nav_msgs/Odometry`
6. 发布 `odom → base_link` TF（`publish_tf:=true` 时）

#### 协方差策略

| 状态 | pose.covariance | twist.covariance |
|------|----------------|-----------------|
| STATUS 新鲜 | `[0.05, 0.05, 0.1]` | `[0.05, 0.1]` |
| STATUS 超时 | `[1.0, 1.0, 3.14]` | `[1000.0, 1000.0]` |

超时后协方差膨胀通知下游（EKF、AMCL）里程计不可信。

#### IMU 发布

`publish_imu_msg(now)` 方法（`publish_imu:=true` 时）：

- `orientation_covariance` 设为 `[1e6, 1e6, 1e6]`（不提供姿态估计）
- `angular_velocity.z` 设为陀螺仪 z 轴（rad/s）
- `angular_velocity_covariance[8]` 设为 `0.02`
- `linear_acceleration` 设为加速度计三轴（m/s²）
- `linear_acceleration_covariance` 设为 `[0.05, 0.05, 0.05]`

EKF 仅使用 IMU 的角速度和姿态，不将加速度作为平移里程计。

## 4. 当前运行参数

| 参数 | 当前值 | 说明 |
|------|--------|------|
| 串口默认参数 | 115200 8N1 | CP2102 USB-UART |
| 控制模式 | `MODE_CLOSED_LOOP`（2） | 上位机固定发送 |
| `cmd_timeout` | 0.25s | `/cmd_vel` 超时后发零速 |
| `drive_keepalive_sec` | 0.10s | 控制帧保活间隔 |
| `control_hz` | 20Hz | 控制循环频率 |
| `status_hz` | 20Hz | 状态轮询频率 |
| `status_timeout` | 0.75s | STATUS 超时后里程计置零 |
| `startup_settle_sec` | 0.20s | 串口打开后等待稳定 |
| `wheel_track_width` | 0.1250m | 轮距 |
| `odom_linear_deadzone` | 0.01 m/s | 线速度死区 |
| `odom_angular_deadzone` | 0.03 rad/s | 角速度死区 |

时序对齐原则：

- `control_hz=20Hz` 对齐下位机 `CTRL_PERIOD_MS=50ms`
- `drive_keepalive_sec=0.10s` 保证失去 `/cmd_vel` 后最晚 0.10s 内重发
- `cmd_timeout=0.25s` 配合 20Hz 控制循环，保证失去 `/cmd_vel` 后约 0.30s 内发出显式零速

## 5. 发布的话题

| 话题 | 消息类型 | 发布条件 |
|------|----------|----------|
| `/odom` | `nav_msgs/Odometry` | 每次 control_loop（20Hz） |
| `/tf` | `geometry_msgs/TransformStamped` | `publish_tf:=true` 时 |
| `/imu/data` | `sensor_msgs/Imu` | `publish_imu:=true` 时 |
| `/chassis/status` | `std_msgs/UInt32` | 每次收到 STATUS |
| `/battery_state` | `sensor_msgs/BatteryState` | 每次收到 STATUS |
| `/motor/left_current` | `std_msgs/Float32` | 每次收到 STATUS |
| `/motor/right_current` | `std_msgs/Float32` | 每次收到 STATUS |

## 6. 状态超时处理

当 STATUS 帧超过 `status_timeout`（0.75s）未更新时：

- 里程计速度置零（位姿保持不变，不回退到命令速度）
- 协方差膨胀至 `[1.0, 1.0, 3.14, 1000.0, 1000.0]`
- 不触发重新探测（串口可能仍在，只是下位机未响应）

## 7. 非闭环模式告警

当 `control_mode ≠ MODE_CLOSED_LOOP` 且已收到过 `/cmd_vel` 时，每 2 秒记录一次告警日志。下位机处于 IDLE 或 OPEN_LOOP 模式时不会执行上位机的速度命令。

## 8. 故障排查

| 现象 | 排查方向 |
|------|----------|
| 节点启动后日志 `No CP2102 serial port detected` | 检查 USB 连接和 `dialout` 组权限 |
| 日志 `Detected CP2102 ports but none replied` | 底盘未上电或串口被占用 |
| `/odom` 不更新 | 检查 STATUS 帧是否正常返回（`status_summary` 日志） |
| 运动方向与命令不一致 | 检查 `odom_angular_sign` 参数 |
| `/odom` 有话题但反馈不变 | 底盘可能处于非闭环模式 |
| 里程计漂移明显 | 调整 `odom_linear_scale`、`odom_angular_scale` |

详细排障见 [07-运维与排障](./07-运维与排障.md)。

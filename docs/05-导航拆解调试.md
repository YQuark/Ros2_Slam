# 导航拆解调试

这份文档是诊断页，不是日常主启动页。日常使用优先执行 `./robot.sh navigation ...`，这里只在链路异常时把导航拆成可验证的小环节。

## 0. 上位机控制底盘

只启动 `stm32_bridge`，不启动雷达、AMCL、Nav2、RViz。

```bash
cd /home/robot/ros2_ws/launch_scripts
./robot.sh base-test --rotate-only
```

期望现象：

- 车原地左转一小段，再原地右转一小段
- 日志中连续出现 `cmd_vel_rx`、`drive_tx`、`status_summary`
- `status_summary` 中的 `odom=(...,...)` 和 `enc_cps=(...)` 在运动时不应一直为 0
- `Cmd log` 中会输出 `SUMMARY rotate_left/rotate_right`，可直接看 `yaw_delta_deg`、`peak_wz_deg_s`、`samples`

如果原地转向稳定，再测直线：

```bash
./robot.sh base-test --with-linear
```

如果你要量化“命令角速度和实际角速度差了多少”，直接跑角速度扫描：

```bash
./robot.sh base-test --calibrate-angular
```

也可以临时改扫描档位，例如：

```bash
BASE_TEST_ANGULAR_SERIES=0.5,1.0,1.5,2.0,2.5 ./robot.sh base-test --calibrate-angular
```

`Cmd log` 里会多出 `CALIBRATION ...` 行，重点看：

- `cmd_wz_deg_s`: 发给底盘的角速度命令
- `effective_wz_deg_s`: 根据 `/odom` 算出的实际平均角速度
- `response_ratio`: 实际角速度 / 命令角速度

这个结果可以直接用来判断：

- 底盘从哪个角速度开始真正克服静摩擦
- 左转和右转是否明显不对称
- 当前导航参数里的最小有效角速度应该设到什么量级

如果标定结果像下面这样：

- 小角速度几乎不动
- `1.0 ~ 2.0 rad/s` 才进入相对可用区
- 左右转明显不对称
- 停止阶段还有较大 `yaw_delta_deg`

那就不要再按“理想差速底盘”的思路调 Nav2。更合适的策略是：

- 抬高 `min_speed_theta`，避开无效小转向
- 降低控制频率和采样密度，减少细碎抖动命令
- 放宽 `yaw_goal_tolerance`，不要在终点附近反复小角修正
- 只把 `1.0 ~ 1.8 rad/s` 这类真正能执行的角速度区间当成主要工作区

如果方向反了，先不要改 Nav2，先修底盘层的线速度、角速度方向或里程计符号。
如果 `SUMMARY ... status=NO_ODOM`，先不要评价控制效果，说明测试阶段根本没收到有效 `/odom`。

## 1. 里程计和 TF

底盘能听 `/cmd_vel` 后，再确认 `/odom` 和 TF：

```bash
source /opt/ros/foxy/setup.bash
source /home/robot/ros2_ws/install/setup.bash
ros2 topic echo /odom --once
ros2 run tf2_ros tf2_echo odom base_link
```

期望：

- 车向前推或发前进命令时，`odom.pose.pose.position.x` 增加
- 原地左转时，yaw 方向和实际一致
- `odom -> base_link` 持续可查

## 2. 雷达和外参

只确认雷达点云跟车体方向一致：

```bash
cd /home/robot/ros2_ws/launch_scripts
./robot.sh sensor lidar
```

另一个终端：

```bash
source /opt/ros/foxy/setup.bash
source /home/robot/ros2_ws/install/setup.bash
ros2 topic hz /scan
ros2 run tf2_ros tf2_echo base_link laser_frame
```

期望：

- `/scan` 频率稳定
- RViz 中车前方障碍物显示在 `base_link` 前方
- 在 `Fixed Frame=base_link` 且只看 `LaserScan` 时，左侧障碍物显示在左侧

如果这里出现左右镜像，优先处理激光坐标链，不要先改 odom：

- 先检查 `src/robot_bringup/config/ydlidar_X2_mapping.yaml` 中的 `inverted`
- 再测试 `--lidar-inverted` / `--no-lidar-inverted`
- 必要时再测试 `--lidar-reversion` 或 `--lidar-yaw-*`
- 错误手性下建出的旧地图直接废弃

## 3. 手动建图和保存

底盘、里程计、雷达都稳定后再建图：

```bash
cd /home/robot/ros2_ws/launch_scripts
./robot.sh mapping lidar --real-base --ekf-base
MAP_PADDING_METERS=1.5 ./robot.sh save-map my_map
```

## 4. AMCL 定位

如果标准导航流程异常，再单独拉起定位链，不启动 Nav2 控制：

```bash
cd /home/robot/ros2_ws/launch_scripts
./robot.sh navigation /home/robot/ros2_maps/my_map.yaml --real-base --ekf-base --localization-only
```

RViz 中使用 `2D Pose Estimate` 设置当前位置。期望激光点和地图墙面重合。

## 5. 全局规划

AMCL 对齐后再开 Nav2：

```bash
cd /home/robot/ros2_ws/launch_scripts
./robot.sh navigation --nav2-only
```

`--nav2-only` 只是回退模式。启动前脚本会先检查 `/map_server`、`/amcl` 是否 active，并等待 `/amcl_pose`。

RViz 中用 `2D Goal Pose` 点目标。先只看 `/plan` 是否生成，不急着评价车走得好不好。

## 6. 局部控制

如果 `/plan` 有路线但车不按路线走，再查：

```bash
source /opt/ros/foxy/setup.bash
source /home/robot/ros2_ws/install/setup.bash
ros2 topic hz /cmd_vel
ros2 topic echo /cmd_vel
ros2 topic echo /local_plan
```

这一步只判断 Nav2 是否持续给底盘发速度，以及局部轨迹是否合理。

## 7. 面板点击导航

前面 0-6 都稳定后，再做面板按钮或网页点选。面板本质上只需要发布 `NavigateToPose` action 或 `/goal_pose`，不要在底盘、定位、规划还没稳定时先做上层面板联调。

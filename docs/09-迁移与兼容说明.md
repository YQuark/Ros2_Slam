# 迁移与兼容说明

这份文档用于说明旧入口和新入口的关系。

## 1. 当前正式入口

统一推荐入口是：

```bash
cd /home/robot/ros2_ws/launch_scripts
./robot.sh <command>
```

最常用的是：

- `mapping`
- `navigation`
- `save-map`
- `doctor`
- `check lidar`
- `stop`

## 2. 旧脚本现在的定位

仓库里保留的 `start_*.sh` 现在只承担兼容转发，不再维护独立启动逻辑。

对应关系：

- `start_mapping.sh` -> `robot.sh mapping`
- `start_navigation.sh` -> `robot.sh navigation`
- `start_nav_fake.sh` -> `robot.sh navigation --fake-base`
- `start_lidar.sh` -> `robot.sh sensor lidar`
- `start_camera.sh` -> `robot.sh sensor camera`
- `start_controller.sh` -> `robot.sh base`
- `start_full_system.sh` -> `robot.sh full`

## 3. 为什么要迁移

旧体系的问题主要有：

- 入口太多
- 参数在多个脚本里重复
- 文档和入口不一致
- 同一能力有多套平行实现

统一到 `robot.sh` 和 `system.launch.py` 后，维护成本明显下降。

## 4. 当前迁移原则

- 新文档只写 `robot.sh`
- 旧脚本只作为兼容层说明
- 两阶段导航只作为回退流程
- 新功能优先补到 `robot.sh` 和 `system.launch.py`

## 5. 迁移检查点

如果你改过工程，至少确认：

- `robot.sh` 还能直接完成建图、保存地图和导航
- `system.launch.py` 参数仍和 `robot.sh` 对得上
- 兼容脚本仍然只是转发
- 根目录 `README.md`、`docs/` 和 `launch_scripts/README.md` 的入口说明一致

# 迁移与兼容说明

本文档说明旧入口与新入口的对应关系、跨平台迁移要点和协议升级历史。

## 1. 统一入口

当前推荐入口：

```bash
cd /home/robot/ros2_ws/launch_scripts
./robot.sh <command>
```

常用子命令：`mapping`、`navigation`、`save-map`、`doctor`、`check`、`stop`。

## 2. 旧脚本兼容层

仓库中保留的 `start_*.sh` 仅承担兼容转发，不再维护独立启动逻辑：

| 旧脚本 | 新命令 |
|--------|--------|
| `start_mapping.sh` | `./robot.sh mapping` |
| `start_navigation.sh` | `./robot.sh navigation` |
| `start_nav_fake.sh` | `./robot.sh navigation --fake-base` |
| `start_lidar.sh` | `./robot.sh sensor lidar` |
| `start_camera.sh` | 已移除（摄像头不属于当前硬件） |
| `start_controller.sh` | `./robot.sh base` |
| `start_full_system.sh` | `./robot.sh full` |

## 3. 迁移原因

旧体系的问题：

- 入口脚本过多，维护成本高
- 参数在多个脚本中重复定义
- 文档和入口不一致
- 同一能力有多套平行实现

统一到 `robot.sh` + `system.launch.py` 后，入口唯一、参数集中、文档一致。

## 4. 迁移原则

- 新文档只写 `robot.sh`
- 旧脚本仅作为兼容层说明
- 两阶段导航仅作为回退流程
- 新功能优先补到 `robot.sh` 和 `system.launch.py`

## 5. x86 → arm64 迁移要点

从旧 x86/Ubuntu 20.04 环境迁移到树莓派 4B arm64/Ubuntu 22.04 时的关键变更：

### 环境变更

| 项目 | 旧环境 | 新环境 |
|------|--------|--------|
| 架构 | x86_64 | aarch64 |
| 操作系统 | Ubuntu 20.04 | Ubuntu 22.04 LTS |
| ROS2 版本 | Foxy | Humble |
| Python | python3.8 | python3.10 |
| 工作空间 | 可变 | `/home/robot/ros2_ws` |

### 构建差异

- `colcon build --symlink-install` 在 arm64 下构建时间显著更长（RPi 4B 约 5-15 分钟）
- `robot_localization` 包在 arm64/Humble 下可能存在兼容问题，建议安装系统包：`sudo apt install ros-humble-robot-localization`
- 部分第三方包可能需要重新编译

### 性能约束

| 约束 | 影响 |
|------|------|
| 4GB RAM | SLAM 栈内存限制 16MB |
| 4×Cortex-A72 @ 1.5GHz | AMCL 粒子数上限 1000 |
| 80°C 降频 | 长时间运行需监控温度 |
| microSD 存储 | 地图和 bag 文件不宜过大 |

### 常见问题

树莓派 4B 雷达迁移的完整问题清单见 [10-树莓派4B雷达迁移问题清单](./10-树莓派4B雷达迁移问题清单.md)，包括：

- 串口权限（`dialout` 组）
- 设备名不稳定（udev 规则）
- Python 可执行权限
- SLAM Toolbox `min_laser_range` 警告
- `robot_localization` 安装层不完整

## 6. Foxy → Humble 迁移要点

### API 变更

- `launch` 子模块路径变更
- `DeclareLaunchArgument` 的 `description` 参数在 Humble 中可选
- `rclpy.node.Node` 的 `declare_parameter` 行为略有差异

### 消息类型

- 大部分消息类型在 Foxy 和 Humble 间兼容
- `sensor_msgs/BatteryState` 在 Humble 中字段更完整

### 构建系统

- `setup.py` 中 `install_requires` 应使用 `setuptools`
- `package.xml` 格式版本应为 `3`

## 7. 协议升级历史

### 旧协议（已废弃）

- 帧格式：COBS 编码 + CRC16 校验
- 状态帧包含：`v_est`、`w_est`、`yaw_est`、`imu_valid`
- 控制帧使用 Q15 定点数编码速度
- 命令字：`MSG_CMD_GET_STATUS`

### 当前 v1 协议

- 帧格式：`0xA5 0x5A + length(1B) + cmd(1B) + payload + checksum8(1B)`
- 状态帧：47 字节，包含轮速、编码器、电池、电流、IMU 原始值、error_flags、control_mode
- 控制帧：`float linear_x + float angular_z + uint8 enable + uint8 mode`
- 命令字：`CMD_SET_VELOCITY`(0x01)、`CMD_ESTOP`(0x02)、`CMD_STATUS`(0x81)

### 已移除的参数

以下参数在 v1 协议中不再有效：

| 参数 | 原因 |
|------|------|
| `use_status_yaw` | v1 协议无 `yaw_est` 字段 |
| `status_yaw_mode` | 同上 |
| `status_yaw_jump_reject_deg` | 同上 |
| `odom_feedback_source` | v1 仅使用轮速计算里程计 |
| `encoder_cpr` | MCU 直接计算轮速 |
| `auto_closed_loop_on_cmd` | 上位机固定发送 closed_loop |
| `max_linear` | 速度限制由下位机控制 |
| `max_angular` | 同上 |

## 8. 迁移检查点

修改工程后至少确认：

- `robot.sh` 能完成建图、保存地图和导航
- `system.launch.py` 参数与 `robot.sh` 对得上
- 兼容脚本仍仅做转发
- `docs/README.md` 和 `README.md` 的入口说明一致
- `colcon build --symlink-install` 构建通过

# 开发与二次复用

这份文档面向需要改工程的人，而不是只想使用工程的人。

## 1. 复用原则

当前工程的复用边界很明确:

- 运维层只保留一个统一入口: `launch_scripts/robot.sh`
- 编排层只保留一个统一 launch: `src/robot_bringup/launch/system.launch.py`
- 具体能力拆到各自 launch 和 package

## 2. 新增能力时先改哪里

如果你要新增一类模式，优先改:

1. `system.launch.py`
2. `robot.sh`
3. 对应的包级 launch
4. 文档

不要先在外层继续叠新的 shell 脚本。

## 3. 新增传感器的思路

新增传感器通常要处理:

- 设备启动
- 参数文件
- 话题命名
- 是否接入 `slam_toolbox`
- 是否接入 RViz

如果这个传感器要参与建图，最终最好还是在 `system.launch.py` 里编排，而不是单独搞一套平行入口。

## 4. 新增模式的思路

工程现在已经有:

- `mapping`
- `navigation`

如果你要再加模式，先问自己:

- 这是不是可以作为 `system.launch.py` 的一个新 `mode`
- 还是只是现有模式的一组参数变化

大多数情况下，后者更合适。

## 5. 参数设计建议

- 参数名保持稳定
- 默认值写在 launch 里
- 用户入口只传关键参数
- 兼容旧参数时保留过渡层，但不要无限期堆积

## 6. 文档复用建议

新增能力时，优先补:

- `02-系统架构.md`
- `07-运维与排障.md`
- `09-迁移与兼容说明.md`

这样可以避免说明散落在脚本和代码注释里。

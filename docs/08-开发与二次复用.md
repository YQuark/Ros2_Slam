# 开发与二次复用

本文档面向需要修改或扩展工程的开发者。

## 1. 复用边界

工程的分层边界明确：

| 层级 | 入口 | 职责 |
|------|------|------|
| 运维层 | `launch_scripts/robot.sh` | 用户意图 → 标准命令 |
| 编排层 | `src/robot_bringup/launch/system.launch.py` | 模式组合 → 节点组装 |
| 包级 | 各包的 `launch/` 目录 | 单包启动逻辑 |
| 参数 | `src/robot_bringup/config/` | YAML 参数文件 |
| 第三方 | `src/third_party/` | vendored 依赖 |

## 2. 新增能力的修改顺序

新增一类模式或功能时，按以下顺序修改：

1. `system.launch.py` — 添加新的模式参数或节点
2. `robot.sh` — 添加对应的子命令或参数
3. 包级 launch — 新建或修改
4. 文档 — 更新 `02-系统架构.md`、`07-运维与排障.md`

不建议在外层新增平行的 shell 脚本入口。

## 3. 新增传感器

新增传感器通常需要处理：

- 设备驱动启动（launch 文件）
- 参数文件（`config/` 目录）
- 话题命名和 remapping
- 是否接入 `slam_toolbox` 或 Nav2
- RViz 配置更新

如果传感器参与建图或导航，应在 `system.launch.py` 中编排，而非独立入口。

## 4. 新增模式

工程当前支持 `mapping` 和 `navigation` 两种模式。新增模式时评估：

- 是否可作为 `system.launch.py` 的新 `mode` 值
- 是否仅是现有模式的参数组合

大多数情况下后者更合适。例如自主探索建图通过 `auto_mapping_drive:=true` 参数扩展，而非新增模式。

## 5. Git 提交规范

本项目提交使用标准前缀首字母大写 + 中文标题 + 中文说明的格式。

### 标题格式

```text
Prefix: 中文提交标题
```

### 允许的前缀

| 前缀 | 用途 |
|------|------|
| `Feat:` | 新增功能 |
| `Fix:` | 修复问题 |
| `Refactor:` | 重构代码 |
| `Chore:` | 工程配置、依赖、脚本 |
| `Docs:` | 文档修改 |
| `Style:` | 代码格式调整 |
| `Test:` | 测试脚本或用例 |
| `Perf:` | 性能优化 |
| `Build:` | 构建系统修改 |
| `Ci:` | 持续集成 |
| `Revert:` | 回退提交 |

### 提交说明要求

说明必须使用中文，包含以下内容：

1. 本次修改了什么
2. 为什么修改
3. 是否影响树莓派部署
4. 是否影响雷达、底盘串口、TF、odom、Nav2 或建图链路
5. 是否需要同步修改下位机协议
6. 已执行哪些验证命令

### 提交示例

```text
Refactor: 重构 STM32 bridge 以适配 v1 协议

- 移除旧 COBS+CRC16+Q15 协议代码
- 新增 v1 帧格式（0xA5 0x5A + checksum8）
- 新增 47 字节 STATUS 帧解析
- 本次修改需要与下位机协议同步
- 已验证 colcon build 通过
```

### 禁止的提交标题

`update`、`fix`、`test`、`修改`、`临时提交` 等无意义标题。

## 6. 分支策略

| 分支 | 用途 |
|------|------|
| `main` | 主分支 |
| `ARM` | 树莓派 4B 上位机工作分支（当前活跃） |
| `feature/*` | 功能分支 |

从 `main` 创建 `ARM` 分支：

```bash
git checkout -b ARM main
```

## 8. 新增 ROS2 包的模板

### 包结构

```text
src/my_package/
├── my_package/
│   ├── __init__.py
│   └── my_node.py
├── launch/
│   └── my_launch.py
├── config/
│   └── my_params.yaml
├── package.xml
├── setup.py
└── setup.cfg
```

### package.xml 要点

```xml
<?xml version="1.0"?>
<package format="3">
  <name>my_package</name>
  <version>0.1.0</version>
  <description>Package description</description>
  <maintainer email="dev@example.com">Author</maintainer>
  <license>MIT</license>
  <buildtool_depend>ament_python</buildtool_depend>
  <exec_depend>rclpy</exec_depend>
  <entry_points>
    <console_scripts>
      <executable_name = my_package.my_node:main</executable_name>
    </console_scripts>
  </entry_points>
</package>
```

### 集成到 system.launch.py

1. 在 `system.launch.py` 中添加 `IncludeLaunchDescription` 或 `Node`
2. 通过 `DeclareLaunchArgument` 添加控制参数
3. 在 `_validate_and_compose()` 函数中添加条件逻辑

## 9. 参数设计建议

- 参数名保持稳定，避免频繁重命名
- 默认值写在 launch 文件中
- 用户入口（`robot.sh`）只传关键参数
- 兼容旧参数时保留过渡层，但设置明确的废弃时间

## 10. 文档维护

新增能力时优先更新：

- `docs/02-系统架构.md` — 节点、话题、TF 树
- `docs/07-运维与排障.md` — 新增检查命令或排查流程

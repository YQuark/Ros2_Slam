# 串口互斥防护需求

## 目标

修复 `robot.sh mapping lidar --real-base --ekf-base` 在雷达健康检查后仍让雷达和 STM32 底盘共用同一个 `/dev/ttyUSB*` 的问题。

## 问题证据

- 运行日志显示健康检查阶段雷达使用 `/dev/ttyUSB0`。
- 正式启动阶段同时打印 `雷达串口: /dev/ttyUSB0` 和 `底盘串口: /dev/ttyUSB0`。
- `stm32_bridge` 随后出现串口读失败和多进程占用提示。
- `slam_toolbox` 持续丢弃 `laser_frame` 消息，地图不随运动更新。

## 验收标准

- 雷达口确定后，真实底盘模式必须排除该雷达口。
- 如果默认底盘口与雷达口冲突，脚本必须主动探测 STM32 串口。
- 如果用户显式指定了与雷达相同的 `--base-port`，脚本必须报错退出。
- 如果无法探测到与雷达不同的 STM32 串口，脚本必须报错退出，不得继续以坏链路启动建图或导航。
- 建图、导航、完整系统入口应使用同一套串口互斥逻辑。

## 非目标

- 不修改 ROS topic、frame_id、package 名称。
- 不修改雷达驱动参数和 STM32 协议。
- 不在本地挂载目录运行 ROS 或硬件验证。

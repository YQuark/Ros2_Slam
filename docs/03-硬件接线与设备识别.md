# 硬件接线与设备识别

这份文档说明当前工程如何识别底盘、雷达和相机，以及为什么现在用自动探测而不是硬编码串口。

## 1. 设备清单

- 底盘下位机: STM32 + CP2102 串口
- 激光雷达: YDLIDAR X2
- 深度相机: Astra Pro

## 2. 推荐接线原则

- 底盘和雷达尽量使用稳定供电
- 不要依赖 USB 口号固定不变
- 不要同时启动多个会抢占同一串口的进程

## 3. 自动识别逻辑

### 底盘

`launch_scripts/detect_base_port.sh` 会先找 `CP2102/cp210x` 特征，再通过底盘协议做主动探测，确认某个串口真的是下位机。

### 雷达

`launch_scripts/detect_lidar_port.sh` 会在现有串口中排除底盘口，再把剩余设备用于雷达。

这意味着当前工程的设备识别逻辑是“按协议和特征识别”，不是“看到 `ttyUSB0` 就当雷达”。

## 4. 你应该如何理解口位变化

- `ttyUSB0` 和 `ttyUSB1` 可能会互换
- `by-id` 可能存在不稳定或重复特征
- 脚本层已经尽量避免人工记口位

## 5. 现场检查

```bash
cd /home/robot/ros2_ws/launch_scripts
./robot.sh doctor
```

如果你只想看底盘:

```bash
./robot.sh base
```

如果你只想看雷达:

```bash
./robot.sh sensor lidar
```

如果你想先检查雷达健康度:

```bash
./robot.sh check-lidar
```

## 6. 常见错误

- 把底盘串口当成雷达串口
- 雷达和底盘都连着，但脚本仍然写死口位
- 串口被旧进程占用
- USB 线材不稳定导致雷达数据报错
